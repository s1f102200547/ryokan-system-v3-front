# Error Handling Guide

### 4段階
    エラー処理の直列のパイプライン
    1.検出
        1.処理前に確認(zod)
        2.処理中に確認(Try-Catch/Result型)
        3.処理後に確認(直後ではなくて一定周期のヘルスチェックなど)
        の３つがある
    2.伝搬
        - 検出したエラーを適切な層(処理)に届ける役割
        - 届ける時に元のエラー情報が欠落しないようにするのが大事
        1.その層では何もしない(他の部分でエラー処理する)場合
        2.その層でエラーを投げる場合
        3.検出した失敗が致命的ではないから記録だけしておいて処理は進める場合
        の3つがある
    3.回復(エラー検出した後、システムを正常な状態に戻そうとする)
        - リトライ
        - フォールバック（代替手段への切り替え）
        - 部分成功の巻き戻し
            複数段階の処理を行う時、途中で失敗したら前の段階で成功した処理をなかったことにする(巻き戻す)必要がある
    4.通知
        回復できなかったものを通知して人間に委ねる
        重要度の区別が大切
        すべてCRITICALにすると無視される、全部INFOにすると重大な障害を見逃す
        - ユーザ向け <- システム情報を見せるのはセキュリティリスクがあるので「あなたが何をすべきか」を伝える
        - 開発者向け <- 「エラーが発生しました」だけでは調査できないので調査に必要な構造化された詳しい情報を記録しておく
        - 運用者向け <- slackなどに通知

    
    



### 主要な手法
    Try-Catch構文(例外を検知して例外を投げる) 
        成功/失敗の2択ではなく失敗(エラー)の種類をしっかり分類するのが大事
        メリット:   コードが短い
        デメリット: どこでエラーが発生したか確認しづらい
                    catchの記述を忘れてもコンパイルが通るので取りこぼすかも
    Result型(関数の返り値を成功/失敗パターンの2種類定義しておく)
        if文を多く書いて失敗のメッセージを分ける必要がある
        メリット:   type(型)定義に沿わない返り値を返せないのでエラーを取りこぼさない
                    FSM設計に適している
        デメリット: コードが長くなる

    リトライとバックオフ
        infraでの外部サービスとの通信では一時的な失敗が頻繁に起きる
        リトライすべきケースとエラーとして諦めるべきケースを区別するのが大切
        
    バリデーション
        Try-CatchやResult型関数にデータを渡す前にチェックする
        今回はinfraと外部サービス間, frontとapi間ででzodを使う
        
    Circuit Breaker
        infraと外部サービスとの通信においてリトライとバックオフをしたうえで諦めてエラーとした場合、
        次からはその特定の外部サービスとの通信はトライせずにエラーを返す。
        ただし、一定時間経ったあとに再度処理要求が来たら通信をトライしてみる
        -> 無駄な通信コストを払わなくて済む

    Dead Letter Queue
        失敗(エラー)を検知できたらdbなどに保管しておいて、後で人間が確認できるようにする
