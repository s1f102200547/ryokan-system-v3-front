# 清掃ボード（v2）

## 概要
清掃スタッフ向けの部屋ごとの清掃状況チェック表。紙出力して使う。

## 業務コンテキスト（なぜこの表が必要か）

掃除スタッフは午前中に来館し、部屋の状態に応じてセットアップを行う：

- **チェックアウト済みの部屋**: 次に宿泊するゲスト情報を基にセットアップ
  - 人数分の布団・アメニティを配置
  - 連泊予定がある場合は「makeup room / do not disturb」カードを机に設置
  - 当日の午後からチェックインするゲストがいなければ翌日以降にチェックインするゲスト情報を下にセットアップ(備考欄に「空室」と書いてある)

- **滞在中の部屋**: 現在滞在中のゲスト情報を基にセットアップ
  - 人数分のアメニティを補充
  - 連泊継続 → カードは机に置いたまま
  - 明日チェックアウト（連泊終了） → カードを押し入れにしまう

- **空室だった部屋**: 既にセットアップ済みなので何もしない

この判断に必要な情報（ゲスト人数・連泊有無・レイトアウト等）をテーブルに集約する。

## v2 -> v3 で変更したいこと
- 日付切り替え: MUIの日付選択コンポーネントをアプリ全体で共有し、清掃ボード・予約状況など各ページが同じ日付を参照する（v2は翌日固定で切り替え不可）
- JSX → TSX
- Architecture.mdに沿って構造を改善する
- 備考欄に、前日が空室の場合は「{部屋番号}: 前日空室のためセットアップ済み」、当日が空室の場合は「{部屋番号}: 本日空室のため次回の予約情報をもとにセットアップ」と記載する。
- 備考欄はテキストボックスみたいにしてユーザが印刷前に記述できるようにしたい。
- v2はあくまで参考で、基本的には踏襲する必要はなく自由に変更して良い。

## v2 -> v3 で踏襲すべきロジック

### テーブル仕様
- 部屋番号は 「21, 22, 31, 32, 42, 43, 61」 の７部屋で固定
- 部屋ごとに1行ずつ。
- A4 で横向きで印刷する
- イメージは {cleaningBoard.png} を参照

### DBスキーマ
- @Schema.md

### 連泊判定
- 過去/当日予約: check_out_date > 対象日+1 → 「連」
- 指定日の: 2泊以上 → 「連」
- ※過去/当日 vs 未来でロジックが異なる。簡略化しないこと

### C/I列の表示
- 当日予約あり → 大人数(子供数)
- 未来予約のみ → カッコ付き表示
- なし → 空欄

### 備考欄の自動生成
- 指定日にその部屋に滞在するゲストがいない場合は「空室」表示
- late_outフラグ && check_out == targetDate+1 → 「レイトアウト11:00」
